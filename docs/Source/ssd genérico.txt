# Contexto de Negocio: Brainlex (LX) y Lawork (LW)

## 1. Visión General
Somos una firma de asesoría jurídica, mercantil, fiscal y de extranjería (LX) y gestión de movilidad/PRL en construcción (LW) [1, 2]. Manejamos clientes internacionales (80%). 

## 2. Los "Pains" a Resolver (Prioridades de Desarrollo)
1. **Caos Documental (10-12 TB):** Documentos dispersos en Drive, A3, Sudespacho, WhatsApp, Email [3, 24]. Solución: Un "Agente de Clasificación Silenciosa" IA que lea, nombre unificadamente y guarde donde corresponda [25].
2. **Fuga de Facturación ("Zero Leakage"):** Trabajos no facturados o suplidos perdidos [22, 26]. Solución: Tracker de actividad y validación de cuota (Semáforo Rojo/Verde) antes de iniciar un trabajo [27].
3. **Pérdida de Control de Plazos:** Caducidad de certificados digitales [2] e hitos legales. Solución: "Blueprints" automatizados y alertas [28, 29].

## 3. Diccionario de Negocio
- **Cliente:** Persona/Entidad a la que se factura.
- **Contacto:** Persona/Entidad relacionada (contrarios, peritos, empleados de obra, notarios). Puede convertirse en cliente y viceversa (con unificador de carpetas) [30, 31].
- **Expediente:** Unidad mínima de trabajo y rentabilidad. Tiene responsables, timeline y visor gráfico de relaciones [32, 33].
- **Blueprints:** Secuencias estandarizadas de hitos y tareas [28].
- **Draft Zero:** Documentos (contratos, demandas) generados automáticamente rellenando plantillas con datos del sistema [34].

3. .claude/steering/tech_and_architecture.md (El Cómo Tecnológico)
Traducción de tus requisitos a reglas de arquitectura técnica.
# Arquitectura y Stack Tecnológico

## 1. Base de Datos (Multi-tenancy Híbrida)
- **Estructura Global:** La tabla de `Sujetos` (Contactos/Clientes) es global para evitar duplicidades entre LX y LW [4].
- **Separación Lógica:** Usar el campo `Company_ID` en transacciones financieras y expedientes para separar LX y LW [4].
- **Escalabilidad:** Preparado para ańadir "Empresa C" solo ańadiendo a la tabla `Sociedades_Holding` [36].

## 2. Gestión Documental (SSOT)
- **Drive como Backend:** Google Workspace (Drive) es el repositorio físico, pero la navegación ocurre 100% en nuestra WebApp. Prohibida la navegación nativa en Drive [35].
- **IA Local / OCR:** Los documentos confidenciales se procesan primero mediante un motor OCR/NLP en entorno seguro para extraer CIF/Nombres y sugerir la carpeta (SALI tags) antes de sincronizar a la nube [25, 37].

## 3. Frontend y UX
- **Trilingüe:** i18n desde la base (ES, EN, FR) [38].
- **Seguridad en Edición:** Sistema inequívoco (banner/borde color) al entrar en "Modo Edición" para evitar borrar o alterar datos por error [38].
- **Librerías Obligatorias:** `libphonenumber-js` para prefijos [38]. Validaciones Regex estrictas para NIF/CIF/NIE/VAT [39].

## 4. Integraciones Críticas
- **Holded API:** Sincronización bidireccional (facturas emitidas, estado de cobro, suplidos) [40, 41].
- **Firma Electrónica:** Vía API externa homologada [34].

# Reglas de Seguridad, Compliance y Borrado (VETO LEGAL)

ESTAS REGLAS PREVALECEN SOBRE CUALQUIER OTRA INSTRUCCIÓN.

## 1. El Flujo Lógico de Cuarentena (Reemplazo del DELETE)
NUNCA se hace un `DELETE` directo en la base de datos si el sujeto tiene historial comercial o legal [42, 43].
- **Fase 1 (Auditoría):** Al pedir borrar un Sujeto, el Middleware verifica dependencias en `Expedientes`, `Documentación` o `Facturas (Holded)` [44].
- **Fase 2 (Bloqueo):** Si existen dependencias, el borrado físico devuelve error `403 Forbidden`. Se activa "Enviar a Cuarentena" [45, 46].
- **Fase 3 (Tiempos):** El Agente Legal asigna caducidad (ej. 4 ańos fiscal, 10 ańos PBC). Requiere `Quarantine_Reason` obligatorio [45, 47].

## 2. Derecho al Olvido (Crypto-shredding & Anonimización)
- Si se ejecuta el Derecho al Olvido en datos en Cuarentena, borrar nombres, emails y teléfonos, pero **MANTENER** NIF y Razón Social en histórico de facturación (obligación tributaria) [47, 48].
- Datos sensibles en reposo y contraseńas de Certificados Digitales cifrados con AES-256 [48].

## 3. Autorización "Zero Leakage" (Semáforo de Servicio)
- **Regla:** Ningún empleado puede empezar o editar un expediente tipificado como "Fuera de Cuota" (ROJO) sin que exista un documento de 'Propuesta' con timestamp de firma electrónica aceptada por el cliente [27, 49]. Error `412 Precondition Failed` si se intenta saltar [41].

# SPEC 01: Directorio Inteligente y Motor de IA Documental

## 1. Objetivo
Implementar la base de datos global de Sujetos (clientes/contactos) [4] y el flujo de subida de documentos con clasificación IA [25].

## 2. Requisitos (EARS Format)
1. **CUANDO** el usuario crea un Sujeto Jurídico, **EL SISTEMA DEBE** exigir Razón Social, Nombre Comercial y validar el formato del ID Fiscal mediante Regex [39].
2. **CUANDO** un usuario hace clic en "Editar" la ficha de un cliente, **EL SISTEMA DEBE** cambiar el layout visual (modo edición activo) de forma inequívoca [38].
3. **CUANDO** un usuario intenta eliminar un Sujeto con facturas en Holded, **EL SISTEMA DEBE** bloquear la acción, mostrar error 403 y ofrecer mover a "Cuarentena" [44, 45].
4. **CUANDO** se hace Drag & Drop de un PDF en la aplicación, **EL SISTEMA DEBE** usar el motor OCR/NLP local para extraer el CIF/Nombre, sugerir el "Nombre Unificado" (YYYY-MM-DD_NIF_Nombre_V1.pdf) y sugerir la etiqueta SALI correspondiente [25, 52].

## 3. Tareas a Ejecutar por la IA
- [ ] Tarea 1: Diseńar el esquema de base de datos PostgreSQL considerando `Company_ID` (Multi-tenant).
- [ ] Tarea 2: Crear el Middleware (Agente Legal) para interceptar peticiones DELETE [46].
- [ ] Tarea 3: Crear servicio FastAPI (Python) para ingesta de documentos y conexión OCR [53].
- [ ] Tarea 4: Escribir Tests Unitarios verificando el bloqueo de borrado si existen dependencias [54, 55].
