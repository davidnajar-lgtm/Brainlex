DOCUMENTO DE ESPECIFICACIONES TÉCNICAS Y FUNCIONALES: ECOSISTEMA BRAINLEX (LX) & LAWORK (LW)

1. ESPECIFICACIÓN TÉCNICA: ARQUITECTURA MULTIENTIDAD Y SUJETOS
1.1. Estructura de la Base de Datos (Multi-tenancy Híbrida)
* Separación Lógica: El sistema debe utilizar una única base de datos pero con un campo Company_ID para separar transacciones financieras y expedientes de LX y LW.
* Directorio de Sujetos Unificado: La tabla de Sujetos (Personas/Empresas) debe ser global para evitar la duplicidad de datos cuando un cliente es común a ambas entidades.
* Escalabilidad: El esquema de base de datos debe permitir la inserción de nuevas empresas (ej. "Empresa C") simplemente ańadiendo un nuevo registro en la tabla de Sociedades_Holding.
* Gestión de Idiomas: La interfaz y los portales de cliente deben ser trilingües (Espańol, Inglés, Francés) desde la base del código.
1.2. Diccionario de Campos: Pestańa "Identidad y Fiscalidad"
El arquitecto debe programar estas validaciones obligatorias para evitar el "desorden" previo:
* Tipología de Sujeto: Selector excluyente entre Persona Física o Persona Jurídica.
o Si es Física: Campos obligatorios: Nombre, Apellido 1, Apellido 2.
o Si es Jurídica: Campos obligatorios: Razón Social, Nombre Comercial.
* Selector de ID Fiscal (Dropdown): Debe incluir validación de formato (Regex) para: NIF, CIF, NIE, DNI, Pasaporte, VAT, TIE, Registro Extranjero y Código Soporte.
* Campos de Control Fiscal (Premium):
o CNAE e IAE: Numérico y descriptivo.
o Régimen IVA: General, Exento, Prorrata % (campo numérico adicional si se activa Prorrata).
o Estado Censal: Fecha de última comprobación en AEAT.
1.3. Lógica de Navegación y Localización
* Interfaz Trilingüe: Implementar una librería de internacionalización (i18n) que cargue archivos JSON de idiomas para Espańol, Inglés y Francés.
* Validación de Teléfonos: Integrar la librería libphonenumber-js para gestionar prefijos internacionales y formateo automático según el país seleccionado.
* Sistema de Edición Inequívoco: Al hacer clic en "Editar", el layout debe cambiar el color del borde o mostrar un banner de "Modo Edición Activo" para evitar cambios accidentales.
1.4. Flujo de Autenticación y Seguridad
* Recuperación de Acceso: Implementar obligatoriamente el flujo de "Olvidé mi contraseńa" con envío de token temporal al email registrado.
* Reglas de Formato de Nombre: El backend debe permitir espacios y caracteres especiales legales (ej. "Lexconomy SLP") pero sanitizar entradas para evitar inyecciones de código.

2. REQUERIMIENTO PARA EL AGENTE SSD (ACCIÓN)
ESTRUCTURA DE AGENTES DE DESARROLLO (SSD AGENTS)
Para evitar los errores de "congelación" y "desorden" de intentos previos, se establece la siguiente jerarquía de trabajo:
1. Agente Orquestador (Project Manager / Architect)
* Función: Es el único con visión global del proyecto (LX + LW).
* Responsabilidad: Distribuye las tareas entre los demás agentes y asegura que los commits al repositorio sigan el orden lógico y estén especificados de una forma clara para que cualquiera pueda entenderlo.
* Veto: Rechaza cualquier implementación que no cumpla con la escalabilidad para futuras empresas.
3. Agente de Datos y Estructura (Database & Knowledge Specialist)
* Función: Diseńa y protege el modelo de datos de "Sujetos" y la taxonomía de etiquetas.
* Responsabilidad: Asegura que las relaciones de la "Pestańa Estructura" (socios, administradores, participadas) se mantengan íntegras.
* Veto: Impide la creación de campos o etiquetas "aleatorias" que no sigan la taxonomía unificada.
4. Agente de Backend (Logic & Integrations Specialist)
* Función: Construye el motor económico y las integraciones con APIs externas (Holded, Google Drive, Firma Electrónica).
* Responsabilidad: Gestiona el cálculo de rentabilidad, las alertas de vencimiento de certificados y el motor de pre-facturación.
* Veto: Bloquea procesos si detecta fallos en la sincronización bidireccional de datos con Holded.
5. Agente de Frontend & UX (Interface & Multi-language Specialist)
* Función: Desarrolla la interfaz trilingüe y los portales de cliente/empleado con el objetivo con máxima simplicidad
* Responsabilidad: Implementa el Drag & Drop (sistema debe gestionar Event Listeners globales) en todo lo que se pueda implantar, como asignar etiquetas a carpetas, documentos, contactos, etc..de forma bidireccional, el modo edición inequívoco y las validaciones visuales de formularios.
* Veto: Rechaza diseńos que no sean responsivos o que no faciliten la búsqueda rápida de expedientes.
6.- EL "AGENTE LEGAL" (LEGAL COMPLIANCE OFFICER)
Este es el módulo de control jerárquico que valida toda acción en la app para evitar "Red Flags" normativas. Este agente debe actuar como un "Middleware" de validación jerárquica.
* Avisos y gestión de Procesos: analizar un expediente si falta la firma de la hoja de encargo, el mandato SEPA o si el documento de identidad (NIE/NIF) ha expirado.
* Gestión de Cuarentena Legal: Los datos no se borran; se mueven a un estado de "Cuarentena" con un contador de tiempo basado en la prescripción legal (4, 6 o 10 ańos según el tipo de dato) para aquellas organizaciones que tienen documentos o expedientes.
* Derecho al Olvido vs. Obligación Contable: Capacidad de anonimizar datos personales manteniendo los registros fiscales inalterables para auditorías.
* Validación de Consentimientos: Controlará la subida de documentación sensible (datos sanitarios o económicos) si no existe un checkbox de "Aceptación de RGPD y Confidencialidad" marcado y vinculado al ID del sujeto.
2.2. Protocolo de Borrado Seguro y "Cuarentena"
Para evitar la pérdida de información sujeta a inspecciones de Hacienda o requerimientos judiciales:
* Deshabilitación del Borrado Directo: Si una entidad (Sujeto) tiene expedientes asociados, documentos en Drive o facturas en Holded, el botón "Eliminar" se deshabilita y se sustituye por "Enviar a Cuarentena".
* Estados de Cuarentena:
o Cuarentena Obligatoria: Se activa automáticamente por el sistema. El Agente Legal asigna un contador de tiempo basado en la prescripción legal (ej. 4 ańos para fiscal, 10 ańos para blanqueo de capitales).
o Cuarentena Voluntaria: El usuario puede mover una entidad a este estado manualmente, debiendo obligatoriamente dejar una Nota de Justificación del motivo.
* Recuperación de Datos: Solo un perfil con rol de "Administrador" o "Legal Officer" podrá restaurar una entidad desde la cuarentena antes de que expire el plazo de borrado definitivo.
2.3. Derecho al Olvido vs. Integridad Contable
* Anonimización: El arquitecto debe implementar un proceso que, a petición del usuario, elimine nombres, apellidos, emails y teléfonos, pero mantenga los registros de facturas (NIF y Razón Social) para cumplir con la ley tributaria.
2.4. Gestión de Certificados Digitales
* Encriptación: Las contraseńas de los certificados deben almacenarse mediante encriptación AES-256.
* Visor de Estado: Un panel específico que liste todos los certificados del grupo (LX y LW), resaltando en rojo aquellos que requieren renovación inmediata, así como un sistema de avisos.


2.6. PROTOCOLO DE COLABORACIÓN (QA & TESTING)
Para garantizar la estabilidad del sistema, el arquitecto de SSD debe configurar los siguientes disparadores (triggers):
* Validación Cruzada: Antes de finalizar cualquier módulo, el Agente Legal y el Agente de Datos deben emitir un certificado de "Cumplimiento".
* QA Obligatorio: Cada agente debe generar sus propios Tests Unitarios. El sistema no permitirá pasar a la siguiente fase sin una validación manual (UAT) satisfactoria por parte de la dirección.
* Control de Sensitive Files: Ningún agente puede modificar archivos de configuración del servidor o de seguridad sin una doble confirmación del Agente Orquestador

3. ESPECIFICACIÓN TÉCNICA: GESTIÓN DOCUMENTAL E IA DE CLASIFICACIÓN
3.1. Infraestructura de Almacenamiento y Conectividad
* Drive como Backend Maestro: El sistema utiliza Google Drive como repositorio principal, pero la interfaz de usuario es la Web App, eliminando la navegación nativa de Drive para evitar el desorden.
* Migración Progresiva (Legacy Bridge): El arquitecto debe diseńar un "puente" para vaciar progresivamente A3, Sudespacho y carpetas antiguas de Drive. Los archivos solo entran al nuevo sistema tras pasar por el proceso de Renombrado y Etiquetado IA.
* Indexación Local: Para archivos sensibles (datos sanitarios o penales), la IA procesará el contenido en un servidor local para garantizar la privacidad antes de cualquier sincronización cloud.  usar la versión de Google Workspace (Business/Enterprise), que permite cumplir con el RGPD y tiene niveles de seguridad superiores a la versión gratuita. La IA Local al procesar los datos "en casa" antes de subirlos, se ańade una capa de seguridad que la mayoría de despachos no tienen.

* 3.2. El "Agente de Clasificación Silenciosa" (IA)
Este agente elimina la carga de decidir dónde guardar cada archivo.
* Omnicanalidad de Entrada: El sistema debe recibir documentos vía Drag & Drop, WhatsApp, Email o escáner.
* Lógica de Procesamiento (OCR + NLP):
1. Lectura: La IA lee el contenido íntegro del documento.
2. Identificación: Detecta al sujeto (por NIF/CIF o nombre) y el tipo de documento (Demanda, Escritura, Factura de Suplido, NIE).
3. Sugerencia de Ubicación: Propone el Nombre Unificado (ej: 2026-03-03_NIF_NombreSujeto_V1.pdf) y la carpeta de destino según la taxonomía.
4. Validación: El usuario solo confirma con un clic (o arrastra a la etiqueta sugerida).
3.3. Taxonomía Unificada y Sistema de Etiquetas (Universal Labels)
* Jerarquía Estricta: No se permite la creación de etiquetas ad-hoc por parte de empleados. Solo el Agente de Datos gestiona el maestro de etiquetas.
* Categorías de Etiquetado:
o Por Proceso: Facturado, Pendiente, En Cuarentena, URGENTE.
o Por Tipo de Actuación: Demanda, Notificación judicial, Escritura, Contrato, Modelo Tributario.
o Por Departamento: Mercantil, Procesal, Fiscal, Laboral (LW).
3.4. Motor de Firma Electrónica y Generación (Draft Zero)
* Integración de Firma: Todo documento que requiera validez legal (Propuestas, RGPD, SEPA) debe enviarse desde la app para firma biométrica, o como mínimo tener un link a un servicio externo que cumpla los rquisitos.
* Generador de Documentos: El sistema debe extraer datos de la Pestańa Identidad (Nombre, NIF, Domicilio) para rellenar plantillas automáticamente, generando un "Borrador Cero" para revisión humana.

3.5. REQUERIMIENTO PARA EL EQUIPO DE AGENTES SSD (ACCIÓN)

4. ESPECIFICACIÓN TÉCNICA: MOTOR ECONÓMICO Y "ZERO LEAKAGE BILLING"
4.1. Arquitectura del Tracker de Actividad e Imputación
* Monitorización de Activos: El sistema debe detectar qué expedientes o documentos de un cliente están abiertos en el equipo del empleado.
* Sugerencia IA de Tiempos: Al final de la jornada o cierre de tarea, el Agente de Backend debe proponer la imputación: "Has trabajado 1h 15min en la Escritura de LX, żimputamos al Expediente X?". Poder relacionar las tareas realizadas con el tiempo controlado
* Registro de Tareas No Facturadas: Interfaz tipo "bandeja de entrada" para trabajos realizados pero no asociados a una factura o cuota, obligando a su resolución antes del cierre de mes.
4.2. Validación de Cuota y Bloqueo Legal
Este es el control crítico para evitar que los clientes rechacen facturas por falta de aviso previo.
* Semáforo de Servicio:
o VERDE (En Cuota): Tareas recurrentes (contabilidad, declaraciones mensuales) definidas en el contrato del cliente.
o ROJO (Fuera de Cuota): Servicios puntuales (multas, recursos, nuevas sociedades).
* Workflow de Aprobación: Si la tarea es "Roja", el Agente de Backend bloquea la edición del expediente y el Agente de Frontend dispara un pop-up: "Servicio fuera de cuota. Generar propuesta de honorarios".
* Firma de Propuesta: El sistema genera el PDF con el precio, lo envía al móvil del cliente y, solo tras la firma digital, libera el aviso al empleado de que puede empezar a trabajar
4.3. Gestión de Gastos y Suplidos (Recuperación Total)
* Vinculación Obligatoria: No se puede registrar un gasto de la empresa sin asociarlo a un Sujeto (Cliente) o a un Expediente específico.
* Método de Pago y Conciliación: Cada suplido debe marcarse según cómo se pagó (tarjeta empresa, transferencia, efectivo) para cruzarlo con la contabilidad de Holded.
* Alertas de Recobro: Panel para el administrador que muestra suplidos pagados por la empresa que aún no han sido incluidos en ninguna factura al cliente.
4.4. Integración Bidireccional con Holded (Holded es el "Maestro de Verdad" para Hacienda, pero la Web App es la "Maestra de Verdad" para la Operativa.)
* Sincronización de Facturas: Emisión de facturas (aproximadamente 1,500 anuales) desde la App hacia Holded, o proporcionando los campos necesarios via api para que holded pueda emitir las factura.
* Estado de Cobro: La App debe leer de Holded si una factura está "Pagada" o "Pendiente" para informar al empleado antes de realizar nuevos trámites para ese cliente.
* Centros de Coste: Separación automática de ingresos y gastos entre LX y LW dentro de la misma instancia de facturación.

ESPECIFICACIÓN TÉCNICA: OPERATIVA DE EXPEDIENTES Y BLUEPRINTS
5.1. Anatomía del Expediente (Objeto de Control)
El expediente es la unidad mínima de trabajo y rentabilidad. El sistema debe gestionar tanto temas puntuales como recurrentes.
* Control de Acceso y Responsables: Definición de un "Responsable de Expediente" y equipo asignado con permisos de lectura/escritura específicos.
* Historial de Actuaciones: Un log inalterable que registre cada acción, documento subido, correo enviado o decisión tomada dentro del expediente.
* Time-line y Calendario: Visualización gráfica de hitos pasados y futuros con integración de alertas para deadlines críticos.
* Visor de Personas Relacionadas: Un gráfico que muestre a todos los contactos (peritos, contrarios, notarios) vinculados específicamente a ese caso.
* Un cliente es una organización (persona física: Nombre, Apellido1, Apellido2) o jurídica (razón social) a la que le emitimos facturas. Un contacto es (persona física: Nombre, Apellido1, Apellido2) o jurídica (razón social) que está relacionado de alguna forma y que debe ser representado en los visuales gráficos que afectan a dos puntos: relaciones entre empresas (grupo, participada, socio, administrador/a (y su persona representante de la sociedad administradora, etc) así visual gráfico dentro del expediente con los contactos relacionados.
5.2. Motor de Blueprints (Flujos Automatizados)
Los Blueprints no son recordatorios, son motores de estado que orquestan el trabajo.
* Disparadores (Triggers): Un hito se activa automáticamente al cumplir una condición (ej: al subir el archivo etiquetado como "Escritura", se dispara el Blueprint de "Liquidación de Impuestos").
* Asignación por Roles: Las tareas del Blueprint se asignan automáticamente según el rol del empleado (Junior para recopilación, Senior para revisión).
* Condicionales de Seguridad: El Blueprint se detiene/Avisa si el Agente Legal detecta que falta un requisito normativo (ej: no se puede "Presentar Demanda" sin la "Hoja de Encargo Firmada").
5.3. Generador de Documentos "Draft Zero"
Para reducir drásticamente el tiempo administrativo:
* Mapeo de Variables: El sistema extrae automáticamente datos de la "Pestańa Identidad" y "Estructura" (NIF, domicilio, representantes).
* Inserción en Plantillas tanto para documentos como para estructuras de carpetas: Inyección de datos en plantillas Word/PDF predefinidas para generar borradores inmediatos de contratos, demandas o comunicaciones.
* Visor Integrado: Capacidad de previsualizar y editar el documento generado sin salir de la Web App.
5.4. Módulo Especializado CAE y Movilidad (Caso LW)
Dado el volumen de trabajadores extranjeros en construcción para LW:
* Gestión Documental de Riesgos: Control de requisitos de Prevención de Riesgos Laborales (PRL) específicos por obra.
* Sincronización con Plataformas Externas: Capacidad (vía bot o API) de subir la documentación validada a plataformas CAE externas.
* Alertas de Cambio en Obra: Si un empleado cambia de obra, el sistema debe alertar para actualizar su documentación preventiva de forma inmediata.

ESPECIFICACIÓN TÉCNICA: FLUJO LÓGICO DE BORRADO Y CUARENTENA
El arquitecto de SSD debe programar este algoritmo en el backend para cualquier petición de eliminación (DELETE request) sobre la entidad Sujeto:
Fase 1: Auditoría de Dependencias (Trigger Inicial)
Cuando el usuario pulsa "Borrar", el sistema debe ejecutar un escaneo automático en las siguientes tablas:
* Expedientes: żExisten registros vinculados (activos o cerrados)?
* Documentación: żHay archivos en el Drive asociados a este ID?
* Economía: żExisten facturas en Holded o registros de suplidos pagados?
Fase 2: Bifurcación de Decisión (Lógica de Negocio)
* CAMINO A (Borrado Físico Permitido):
o Condición: El Sujeto no tiene expedientes, ni documentos, ni facturas (es un alta errónea o pre-cliente sin actividad).
o Acción: Mostrar pop-up de "Doble Confirmación". Si se acepta, se elimina el registro de la base de datos definitivamente.
* CAMINO B (Bloqueo y Cuarentena Obligatoria):
o Condición: El Sujeto tiene cualquier dato vinculado en las tablas mencionadas.
o Acción: El botón "Borrar" se deshabilita visualmente. Se activa el botón "Enviar a Cuarentena".
Fase 3: Ejecución de la Cuarentena (Agente Legal)
Si se sigue el Camino B, el sistema ejecuta lo siguiente:
1. Cambio de Estado: El campo Status del Sujeto cambia a QUARANTINE.
2. Cálculo de Expiración: El Agente Legal aplica la regla de tiempo según el dato más restrictivo (ej. 10 ańos por Blanqueo de Capitales o 4 ańos por Hacienda).
3. Nota Obligatoria: Si la cuarentena es voluntaria, el sistema bloquea el proceso hasta que el usuario escriba el motivo en el campo Quarantine_Reason.
4. Bloqueo de UI: El Sujeto desaparece del Directorio Activo y solo es visible en el panel de "Administración / Papelera de Cuarentena".
Fase 4: Finalización del Ciclo (Derecho al Olvido)
* Anonimización: A petición del usuario y validación del Agente Legal, se ejecutan scripts de limpieza que borran datos de contacto (email, teléfono, dirección personal) pero mantienen el NIF y Razón Social en los registros históricos de facturación para auditorías.
* Purga Automática: Al cumplirse el contador de tiempo de la cuarentena, el sistema notifica al Agente Orquestador para la destrucción definitiva del registro.

REQUERIMIENTO PARA EL AGENTE DE BACKEND (ACCIÓN SSD)
"Debes implementar un Middleware de Integridad que intercepte toda sentencia de borrado. Si la función checkDependencies(subjectID) devuelve true, la base de datos debe rechazar la eliminación y forzar la actualización del campo deleted_at y quarantine_expiry_date. No permitas que el Frontend salte esta validación".
* Recuperación de Acceso: Implementación de flujos de "Recuperar Contraseńa" y validación de formatos de campo (ej. Lexconomy SLP con espacios permitidos).

8. MIGRACIÓN Y LEGACY
* Conexión Holded: Sincronización bidireccional de estados de factura y cobro.
* Apagado de Software Antiguo: La app servirá de puente para vaciar progresivamente A3 y Sudespacho, moviendo la info al nuevo Drive estructurado hasta que los sistemas antiguos puedan ser anulados, renombrando y ańadiendo descripción o las extracciones de datos que correpondan.
